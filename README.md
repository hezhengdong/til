但是经过分析后发现并不合适：Prometheus 的主要职责是「监控」，监控主要用于查看「当前状态」「近期趋势」，「消息推送成功/失败数」这类数据属于需要长期持久化的「累积值」，而 Prometheus 的设计哲学适合「重启归零」的数据。

因此对于「业务状态追踪」，应当交由 ClickHouse 等 OLAP 数据库。

对于具体的技术实现，尚在探索中，有很多细节需要考量。

---

技术选型时，需要对技术的原理有所了解，才能判断当前业务应当使用什么技术。

数据库分为 OLTP 与 OLAP，二者技术上的本质区别在于行式存储与列式存储。

- OLTP（在线事务处理）为行式存储，例如订单表中订单 ID、用户 ID、金额都挨在一起。
- OLAP（在线分析处理）为列式存储，例如订单表中所有金额存储在一个文件中。

因此 OLTP 适用于随机读写、以行为单位的查询（`Select *`），如用户增删改查自己的订单；OLAP 适用于批量写入，以列为单位的查询，如查询昨日总销售额。

TSDB（时序数据库）是一种特殊的 OLAP。

- OLAP 以「列」为基本单位，所有事件混在一起。
- TSDB 以「时间线」为基本单位，例如 Prometheus 监控一台机器，为每个指标维护一条时间线，监控**某个指标随时间如何变化**。

因此 TSDB 适用于系统监控（CPU、内存）、IoT 传感器、金融行情；OLAP 适用于用户行为分析、业务明细统计、业务报表。

此外还有 HTAP，粗浅了解认为是通过分布式的副本机制实现同时提供列式存储与行式存储的效果，通过 SQL 分析将不同需求分发到不同的副本下。还有 HDFS 和 Hive，目前理解 HDFS 是大型的 Linux 文件系统，Hive 是给 HDFS 加了一个 SQL 壳子。

### 物化视图

数据库中，普通视图指的是通过 SQL 构造一个虚拟的表。运行基于普通视图的 SQL 语句，普通视图会将其翻译为基于底层数据库表的查询语句。

自己尝试将 ClickHouse 当作 Kafka Consumer 时，AI 给出的实现方案为 Kafka 引擎表 -> 物化视图 -> 真正的数据库表。

普通视图只是底层表的翻译层，但是物化视图会将数据进行持久化。在自己的业务实现中，物化视图就像一个中转站，将数据持久化在数据库表中。

**什么是 Shared Nothing、Shared Everything？**

仅从概念上理解：Shared Nothing 就像多个小型 PC，每台 PC 的硬盘、内存、CPU 都不是共享的；Shared Everything 就像是一个大型 PC，共享一个大硬盘、大内存、大 CPU。

从存算一体和存算分离的角度理解：

- 从前大家构建数据库时，都是基于多个小型 PC 构建的，每个 PC 的内存硬盘都要充分利用。
- 后来有了 AWS EC2 和 AWS S3 这种基础设施，虽然 EC2 和 S3 的本质都是多个小 PC，但是 AWS 基于这些 PC 实现了上层抽象，EC2 就像是无数个计算单元，S3 就像是一个无限硬盘。
- 直接基于 AWS 实现的 EC2 和 S3 抽象构建数据库，而非基于底层的 PC。数据库的基础设施从 Shared Nothing 变为 Shared Everything。

**什么是云原生？**

按需付费，弹性伸缩。

### 事务

2PL 实现可串行化。其中「两阶段」是关键，事务必须严格分为持有锁和释放锁两个阶段，不能出现先释放锁又持有锁的情况。问题在于性能差。

MVCC 的思想很好理解，跟 Git 分支差不多，为数据维护一个版本链，读写分离提高性能。

MVCC 的工程实现，通过隐藏字段、undo log 版本链、read view 实现。具体可以看一个不错的讲解视频（[link](https://www.bilibili.com/video/BV1hL411479T/)）。

读操作可以分为快照读（普通 select）和当前读（insert、update、delete、select ... for update）。整个事务内，当发生快照读时，一直读取同一个没有被事务操作的数据版本，就解决了脏读和不可重复读的问题。

幻读问题（也是写写冲突问题、当前读问题）需要额外解决。比如数据库中有张三，事务 A B 读张三，事务 A 把张三改为张四，事务 B 又把张四改为张五，然后事务 B 和事务 A 再读。其中的修改操作属于「当前读」操作，要么通过锁让事务串行，要么回滚其中一个事务。

参考资料：主要是 IT 老齐的视频，DDIA 感觉讲解一般，AI 幻觉太重。

如果未来回看这里发现什么都看不懂了，记得回看 IT 老齐的视频。如果困惑 MVCC 没有实现可重复读，那么需要注意自己是否搞错了「当前读」的概念。「当前读」需要额外处理，而非像「快照读」那样直接执行。

### OS - 进程线程

进程是操作系统分配资源的最小单位，线程是CPU调度的最小单位。

按照自己的理解，进程用于自己的虚拟内存，但是线程没有，线程依附于进程的虚拟内存。

另外，OS 内存本质上也是按照页调度的，就像 MySQL 那样。

### Kafka

Kafka 还有很多关键设计，如 ISR、acks 等。

分区副本数、ISR、acks、min.insync.replicas 间的关系：

- 分区副本数通过参数指定
- ISR 用于维护参与同步的副本数，可能会少于参数指定的副本数
- acks 指定 1 代表只需 leader 写成功即可，all 代表需要 ISR 中所有副本同步成功
- min.insync.replicas 限制 ISR 中副本数至少有多少个

所以生产设置一般是 replication.factor（副本数）= 3、min.insync.replicas = 2、acks = all。

另外：

- enable.auto.commit 建议设置为 false，除非不怕丢消息
- 自己的项目本身是缺功能的， Kafka 生产端也有可能丢消息，同步异步重试这些完全没考虑

### 大数据

起因是 boss 上遇到的实习生测试题，自己对这方面也感兴趣，因此浅尝辄止了解了一下。

从历史演进的视角来看大数据相关知识，也算是 OLAP 相关知识

- 数据增长，单台机器存不下算不完，所以诞生 GFS + MapReduce。
- MapReduce 本质是批处理操作，一次计算需要很长时间。因此诞生流处理操作，并通过 Lambda 架构 Join 两边数据进行展示。
- 后来又有了流批一体，有了 Doris 这种云原生的 OLAP 数据库。

另外，还有很多细碎知识点：

- 自己低估了 MapReduce，比起 MapReduce 这一简单的思想，其「计算逻辑向数据移动」「并非先建表再存数据」的设计思想似乎更加重要。
- MapReduce 的时代背景是磁盘 IO 大于网络，后来网络 IO 超过了磁盘，所以云原生变得可行起来。
- MapReduce 中间阶段要写入硬盘，太慢了，所以 Spark 基于内存计算。就像通过 Hive 做到 SQL 实现 MapReduce 一样，Spark SQL 也是如此。
- BigTable 是 LSM Tree 在工业界大规模应用的典范？
- 目前来看，OLTP 与 OLAP 在技术角度完全南辕北辙。比如分布式层面，OLAP 不需要支持 ACID 事务。
- 自己此前的学习侧重 OLTP，OLAP 在任何层面都是一个全新领域。Doris 底层架构完全看不懂，不像 Spanner 那样了解很多前置知识。

- ETL 如何实现？
- 技术层面如何理解数据湖、湖仓一体？
