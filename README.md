自己要为个人项目添加可观测性，相关知识了解不足，探索了三天。

一个程序的运行状况如何，我认为可以分为两个维度，面向开发/运维的「可观测性」，面向运营的「业务状态追踪」。

对于「可观测性」，核心是指标、日志、链路追踪，IT 老齐的相关视频讲解很清晰。

项目有一个需求是「为运营展示单次推送任务的成功率」，最开始自己想着集成 Prometheus 客户端，将其作为一个指标展示。

但是经过分析后发现并不合适：Prometheus 的主要职责是「监控」，监控主要用于查看「当前状态」「近期趋势」，「消息推送成功/失败数」这类数据属于需要长期持久化的「累积值」，而 Prometheus 的设计哲学适合「重启归零」的数据。

因此对于「业务状态追踪」，应当交由 ClickHouse 等 OLAP 数据库。

对于具体的技术实现，尚在探索中，有很多细节需要考量。

---

技术选型时，需要对技术的原理有所了解，才能判断当前业务应当使用什么技术。

数据库分为 OLTP 与 OLAP，二者技术上的本质区别在于行式存储与列式存储。

- OLTP（在线事务处理）为行式存储，例如订单表中订单 ID、用户 ID、金额都挨在一起。
- OLAP（在线分析处理）为列式存储，例如订单表中所有金额存储在一个文件中。

因此 OLTP 适用于随机读写、以行为单位的查询（`Select *`），如用户增删改查自己的订单；OLAP 适用于批量写入，以列为单位的查询，如查询昨日总销售额。

TSDB（时序数据库）是一种特殊的 OLAP。

- OLAP 以「列」为基本单位，所有事件混在一起。
- TSDB 以「时间线」为基本单位，例如 Prometheus 监控一台机器，为每个指标维护一条时间线，监控**某个指标随时间如何变化**。

因此 TSDB 适用于系统监控（CPU、内存）、IoT 传感器、金融行情；OLAP 适用于用户行为分析、业务明细统计、业务报表。

此外还有 HTAP，粗浅了解认为是通过分布式的副本机制实现同时提供列式存储与行式存储的效果，通过 SQL 分析将不同需求分发到不同的副本下。还有 HDFS 和 Hive，目前理解 HDFS 是大型的 Linux 文件系统，Hive 是给 HDFS 加了一个 SQL 壳子。
